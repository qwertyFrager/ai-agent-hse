# Материалы для отчета по проекту

## 1. Краткая идентификация проекта

- Название проекта: `ai-agent-hse`
- Тип проекта: AI-ассистент учебного офиса на основе RAG
- Предметная область: поиск и объяснение информации из внутренних учебных документов
- Основные каналы доступа:
  - веб-интерфейс
  - HTTP API
  - Telegram-бот

## 2. Проблема, которую решает проект

Проект решает задачу быстрого доступа к информации из локального набора нормативных и организационных документов. В исходной ситуации документы существуют в разных форматах и неудобны для ручного поиска. Пользователю приходится:

- открывать несколько файлов вручную
- искать нужные формулировки по ключевым словам
- интерпретировать длинные регламенты и таблицы

Система автоматизирует этот сценарий: извлекает текст из документов, индексирует его и позволяет задавать вопрос в естественной форме.

## 3. Цель разработки

Разработать систему, которая:

- принимает вопрос в свободной форме
- ищет релевантные фрагменты по локальной базе документов
- формирует понятный ответ
- возвращает ссылки на источники
- работает через несколько пользовательских каналов

## 4. Основные пользователи

- студент
- сотрудник учебного офиса
- администратор, который загружает и обновляет документы
- пользователь Telegram-бота

## 5. Функциональные возможности проекта

### 5.1. Работа с документами

- разбиение документа на чанки

### 5.2. Работа с вопросами

- прием вопросов через Telegram
- поиск релевантных фрагментов
- генерация ответа через LLM
- возврат источников

### 5.3. Веб-интерфейс

- страница пользовательского дашборда
- просмотр ответа и источников
- статистика по документам и чанкам
- история запросов текущего процесса

### 5.4. Администрирование

- вход администратора
- просмотр списка документов
- загрузка нового документа

- удаление документа

### 5.5. Telegram-бот

- кнопка `Документы`

## 8. Причины выбора архитектуры

### 8.1. Почему единая кодовая база

Было выбрано единое приложение с несколькими интерфейсами доступа, потому что:

- логика поиска и генерации ответа должна быть одинаковой в API, вебе и Telegram
- это уменьшает дублирование кода
- проще тестировать и сопровождать систему
- проще реализовать MVP

### 8.2. Почему модель хранения `docs + chunks`

Такая структура выбрана потому, что:

- документ как целое удобно хранить отдельно от его текстовых фрагментов
- метаданные документа можно использовать в поиске
- чанк удобен как единица поиска и контекста для LLM
- переиндексация документа становится простой и управляемой

### 8.3. Почему сначала реализован FTS, а не vector search

- проще реализация
- ниже инфраструктурная сложность
- для небольшого набора документов FTS достаточно эффективен
- система подходит для учебного проекта и может быть позже расширена до hybrid search

## 9. Архитектура системы

### 9.1. Внешние компоненты

- пользовательский браузер
- Telegram-клиент
- HTTP-клиент
- PostgreSQL
- файловая система с документами
- внешний LLM provider

### 9.2. Внутренние компоненты

- `app/main.py` - HTTP API, веб-дашборд, админка
- `app/indexer.py` - индексирование документов
- `app/search.py` - полнотекстовый и fallback-поиск
- `app/rag.py` - основной пайплайн ответа
- `app/llm.py` - клиент OpenAI-compatible API
- `app/models.py` - модели БД
- `app/admin_auth.py` - аутентификация администратора
- `bot/main.py` - Telegram-бот

### 9.3. Текущая контейнерная схема

При запуске `docker compose up --build` поднимаются:

- `postgres`
- `api`
- `bot`

При этом:

- статический веб-интерфейс обслуживается контейнером `api`
- бот и API используют одну и ту же БД
- каталог `docs` монтируется в оба контейнера

## 10. Подробности реализации индексации

### 10.1. Последовательность обработки документа

1. Найти файл в каталоге `docs/`
2. Проверить, поддерживается ли расширение
3. Посчитать checksum
4. Проверить, нужно ли обновлять документ
5. Извлечь текст
6. Нормализовать текст
7. Построить описание документа
8. Разбить текст на чанки
9. Сохранить документ и чанки в БД

### 10.2. Особенности обработки форматов

- PDF читается постранично
- DOCX и DOTX обрабатываются как обычный документ или OpenXML-шаблон
- Excel и CSV переводятся в строковое представление с указанием листов, строк и значений ячеек

### 10.3. Почему это важно

Такой подход делает табличные документы пригодными для текстового поиска и для передачи в LLM как обычного контекста.

## 11. Подробности реализации пайплайна ответа

### 11.1. Общая схема

1. Пользователь задает вопрос
2. Вопрос нормализуется
3. Формируются дополнительные поисковые запросы
4. Выполняется поиск по `chunks`, `title`, `description`
5. Лучшие результаты собираются в контекст
6. Формируется запрос к LLM
7. Возвращается итоговый ответ и список источников

### 11.2. Что получает LLM

- системную инструкцию
- исходный вопрос
- краткий обзор библиотеки документов
- извлеченные фрагменты

### 11.3. Fallback-режим

Если LLM недоступна:

- система не останавливается
- пользователю показываются найденные материалы
- это повышает устойчивость системы

## 12. Реализация Telegram-бота

### 12.1. Зачем он нужен

Telegram-бот делает систему доступной в привычном мессенджере и не требует отдельного веб-интерфейса на стороне пользователя.

### 12.2. Логика работы бота

- бот работает через polling
- использует общий пайплайн `answer_question(...)`
- поддерживает постоянное меню
- умеет показывать список документов
- умеет запускать заранее заданные популярные запросы

### 12.3. Особенность текущей реализации

Telegram-бот не ходит в HTTP API, а напрямую использует общие Python-модули проекта. Это уменьшает задержку и убирает лишний уровень сетевого взаимодействия внутри системы.

## 13. Реализация административной панели

### 13.1. Назначение

Админка нужна для управления корпусом документов без прямой работы с файловой системой и БД.

### 13.2. Что реализовано

- login/logout
- просмотр списка документов
- загрузка файлов
- редактирование метаданных
- удаление документов

### 13.3. Безопасность

Аутентификация реализована через:

- учетные данные из переменных окружения
- подписанную cookie-сессию
- срок действия сессии 8 часов

## 17. Ограничения, которые честно стоит указать в отчете

- история запросов пока не сохраняется в БД
- поиск пока не использует векторные представления
- качество ответа зависит от качества извлечения текста
- внешний LLM-сервис влияет на задержку и надежность
- текущая auth-модель подходит для учебного проекта, но не для промышленной эксплуатации

# Workflow и архитектура проекта

## Назначение системы

Проект представляет собой AI-агента "учебный офис", который помогает пользователю получать ответы по локальной базе документов в свободной форме. Система совмещает два режима:

- `grounded mode`: ответ строится с опорой на найденные фрагменты документов и их метаданные
- `general mode`: если подтвержденных данных в базе недостаточно, агент все равно продолжает диалог и дает полезный общий ответ через LLM

Таким образом, система не ограничивается классическим поиском по базе, а реализует гибридную модель: `retrieval + LLM orchestration`.

## Цели проектирования

При проектировании были заложены следующие цели:

- минимальная и понятная архитектура без микросервисного дробления
- единый backend для API, веб-интерфейса и Telegram-бота
- хранение исходных файлов на диске, а в БД только индекса и метаданных
- идемпотентная индексация документов
- возможность постепенно усилить систему без полного переписывания

## Основные сущности

Система использует две основные таблицы:

- `docs`: описание документа, путь к файлу, тип, checksum, статус и краткое описание назначения документа
- `chunks`: текстовые фрагменты документа, используемые для поиска и передачи контекста в LLM

На уровне предметной модели:

- один документ `docs` содержит несколько чанков `chunks`
- каждый чанк относится ровно к одному документу
- описание документа хранится в `docs.description` и используется как часть retrieval-контекста

## Логика работы системы

### 1. Сценарий индексации

Сценарий индексации запускается через:

- старт FastAPI-приложения
- старт Telegram-бота
- CLI-команду `python -m scripts.index`
- API-эндпоинт `POST /index`

Последовательность обработки:

1. Система обходит каталог `DATA_DIR`.
2. Для каждого поддерживаемого файла вычисляется `checksum`.
3. Если checksum не изменился и описание уже существует, документ пропускается.
4. Если файл новый или изменился, текст извлекается в зависимости от формата:
   - `pdf` через `pypdf`
   - `docx` и `dotx` через `python-docx` или чтение OpenXML
   - `xls`, `xlsx`, `csv` через `pandas`
5. Извлеченный текст нормализуется: убираются лишние пробелы и пустые блоки.
6. Для табличных файлов строки переводятся в более "читаемый" текстовый вид, чтобы улучшить поиск и понимание LLM.
7. На основе текста формируется краткое описание документа:
   - через LLM, если она доступна
   - через эвристический шаблон, если LLM недоступна
8. Текст документа режется на чанки фиксированного размера с overlap.
9. Чанки сохраняются в `chunks`, а для поиска используется `tsvector`.

Результат индексации:

- система получает корпус документов в структурированном виде
- у каждого документа появляется краткое описание "что внутри" и "когда обращаться"
- поиск начинает учитывать не только контент чанков, но и семантически полезные метаданные документа

### 2. Сценарий ответа на вопрос

Ответ на вопрос проходит через единый пайплайн в `app/rag.py`.

Последовательность обработки:

1. Пользователь отправляет вопрос через веб, API или Telegram.
2. Вопрос нормализуется.
3. Из вопроса строится набор поисковых форм:
   - исходный текст вопроса
   - упрощенный набор ключевых терминов без стоп-слов
4. По каждой поисковой форме выполняется retrieval.
5. Retrieval ищет совпадения по:
   - `chunks.content`
   - `docs.title`
   - `docs.description`
6. Лучшие найденные фрагменты объединяются в контекст.
7. Дополнительно формируется краткий обзор каталога документов.
8. В LLM передается:
   - системная инструкция
   - вопрос пользователя
   - обзор локальной библиотеки
   - релевантные фрагменты
9. LLM формирует ответ в одном из двух режимов:
   - с подтвержденными фактами из базы
   - с общей полезной консультацией, если в базе нет точных данных
10. Пользователь получает:
   - связный ответ
   - список источников, если retrieval нашел релевантные документы

### 3. Сценарий работы Telegram-бота

Telegram-бот не содержит отдельной бизнес-логики. Он использует тот же backend-пайплайн, что и HTTP API.

Последовательность:

1. Бот принимает сообщение.
2. При старте он убеждается, что БД и индекс готовы.
3. Вопрос пользователя передается в общую функцию `answer_question`.
4. Ответ и источники отправляются обратно в Telegram.

Это решение уменьшает дублирование и обеспечивает одинаковое поведение веб-интерфейса, API и Telegram-бота.

## Архитектура на уровне компонентов

Ниже приведено логическое разбиение системы на внутренние компоненты.

### Компонент `Interface Layer`

Состав:

- `FastAPI endpoints` в `app/main.py`
- `Telegram bot` в `bot/main.py`
- `Web dashboard` в `app/static/`

Ответственность:

- принимать пользовательские запросы
- запускать индексацию
- возвращать ответы и служебные данные
- обеспечивать каналы взаимодействия с пользователем

### Компонент `Application Layer`

Состав:

- `RAG orchestrator` в `app/rag.py`
- `History manager` в `app/history.py`

Ответственность:

- координировать поиск, сбор контекста и вызов LLM
- различать подтвержденный и неподтвержденный ответ
- сохранять историю запросов текущего процесса

### Компонент `Retrieval Layer`

Состав:

- `Search service` в `app/search.py`
- текстовая нормализация в `app/utils_text.py`

Ответственность:

- выполнять поиск по документам и чанкам
- расширять retrieval за счет `title` и `description`
- подготавливать текст для поиска и chunking

### Компонент `Indexing Layer`

Состав:

- `Indexer` в `app/indexer.py`
- CLI-запуск в `scripts/index.py`

Ответственность:

- обходить файловую систему
- извлекать текст из документов
- описывать документы
- формировать чанки
- поддерживать идемпотентное обновление индекса

### Компонент `LLM Integration Layer`

Состав:

- `LLM client` в `app/llm.py`

Ответственность:

- вызывать OpenAI-compatible API
- использовать LLM как для ответа пользователю, так и для генерации описаний документов

### Компонент `Persistence Layer`

Состав:

- `SQLAlchemy models` в `app/models.py`
- `DB bootstrap` в `app/db.py`
- `PostgreSQL`

Ответственность:

- хранить документы и чанки
- обеспечивать полнотекстовый поиск
- поддерживать простую, устойчивую схему хранения

## Архитектура на уровне контейнеров

На уровне container view систему можно описать так.

### Контейнер 1. Пользовательские клиенты

Состав:

- браузер
- Telegram-клиент
- внешний HTTP-клиент

Роль:

- инициируют запросы к системе

### Контейнер 2. AI Backend

Технологии:

- `Python`
- `FastAPI`
- `SQLAlchemy`
- `python-telegram-bot`

Роль:

- центральный вычислительный контейнер системы
- содержит API, оркестрацию RAG, индексацию, Telegram-интеграцию, дашборд и LLM-вызовы

Внутренние модули контейнера:

- API controller
- Telegram handler
- Indexer
- Retrieval service
- RAG orchestrator
- LLM adapter

### Контейнер 3. PostgreSQL

Роль:

- хранит таблицы `docs` и `chunks`
- обеспечивает полнотекстовый поиск по корпусу документов

### Контейнер 4. Файловое хранилище

Роль:

- хранит исходные документы в каталоге `docs/`
- является источником данных для индексатора

### Контейнер 5. Внешний LLM Provider

Роль:

- генерирует финальный ответ пользователю
- генерирует краткие описания документов

Примеры:

- OpenAI API
- любой OpenAI-compatible endpoint

## Текстовое представление связей между контейнерами

Можно использовать следующую формулировку для отчета или диаграммы:

1. Пользователь взаимодействует с `AI Backend` через браузер, Telegram или HTTP API.
2. `AI Backend` получает данные документов из файлового каталога `docs/`.
3. `AI Backend` сохраняет индекс и метаданные в `PostgreSQL`.
4. `AI Backend` при необходимости обращается к внешнему `LLM Provider`.
5. `AI Backend` возвращает результат пользователю через исходный канал взаимодействия.

## Обоснование архитектурных решений

### Почему выбран единый backend

Выбор единого backend позволил:

- сократить сложность разработки
- избежать дублирования бизнес-логики между API и ботом
- упростить отладку
- оставить архитектуру расширяемой без раннего перехода к микросервисам

### Почему выбрана схема `docs + chunks`

Эта схема дает баланс между простотой и функциональностью:

- `docs` содержит метаданные и описание документа
- `chunks` содержит текст для retrieval
- связь один-ко-многим позволяет переиндексировать документы независимо

### Почему используется always-on LLM

Режим always-on LLM выбран потому, что целевой сценарий ближе к AI-ассистенту, чем к поисковому интерфейсу. Пользователь ожидает:

- живой диалог
- понятные ответы
- помощь даже при неполной информации

При этом retrieval остается важным источником grounded-контекста и ограничивает риск галлюцинаций, когда релевантные документы есть в базе.

## Ограничения текущей версии

На текущем этапе сохраняются следующие ограничения:

- история запросов хранится в памяти процесса и не переживает рестарт
- нет пользовательской памяти между сессиями
- retrieval построен на FTS и метаданных, но не использует embeddings
- релевантность ответа зависит от качества извлечения текста из PDF и табличных файлов
- LLM является внешней зависимостью по сети и может влиять на задержку ответа

## Roadmap

### Этап 1. Стабилизация текущей версии

- доработать README и эксплуатационную документацию
- добавить больше диагностических логов по retrieval и вызовам LLM
- улучшить отображение источников и документов на дашборде
- добавить health-check не только backend, но и подключения к БД и LLM

### Этап 2. Усиление agentic-функций

- добавить память диалога по пользователю
- реализовать хранение контекста сессии
- внедрить query rewriting через отдельный LLM-шаг
- добавить классификацию вопроса по типам: по расписанию, по регламенту, по ВКР, свободный вопрос

### Этап 3. Усиление retrieval

- добавить embeddings и vector search
- реализовать hybrid search: `FTS + semantic retrieval`
- улучшить разбор Excel- и PDF-документов
- добавить reranking найденных чанков перед передачей в LLM

### Этап 4. Масштабирование пользовательских каналов

- добавить авторизацию для административных операций
- вынести Telegram и web notifications в управляемый слой событий
- подготовить backend к интеграции с другими каналами, например web chat widget или VK bot

### Этап 5. Интеллектуальные функции следующего уровня

- подключить внешние источники знаний, например официальный сайт вуза
- добавить агентные сценарии со множеством инструментов
- внедрить автообновление базы знаний по расписанию
- добавить explainability-блок: почему агент дал именно такой ответ

## Краткий вывод для главы проектирования

Разработанная архитектура представляет собой централизованную AI-систему с единым backend-контуром, в котором объединены индексация документов, полнотекстовый поиск, orchestration retrieval-логики и генерация ответов через LLM. Такое решение обеспечивает низкий порог реализации, прозрачную структуру компонентов и возможность поэтапного перехода от минимального RAG-сервиса к полноценному AI-агенту учебного офиса.

# Workflow и архитектура проекта

## 1. Назначение проекта

Проект `ai-agent-hse` реализует AI-ассистента учебного офиса, который отвечает на вопросы по локальной базе документов. Система объединяет:

- индексирование документов из каталога `docs/`
- поиск по содержимому и метаданным документов
- генерацию ответа через OpenAI-compatible LLM
- веб-интерфейс и административную панель
- Telegram-бота

Основная идея проекта: пользователь задает вопрос в свободной форме, а система сначала ищет релевантные документы и фрагменты, после чего строит ответ на их основе. Если подтвержденного контекста недостаточно, система все равно пытается дать полезный ответ, явно отделяя подтвержденные факты от общего объяснения.

## 2. Актуальная схема развертывания

В текущем состоянии проект поддерживает два режима запуска.

### Локальный режим

- Postgres запускается отдельно
- API запускается командой `uvicorn app.main:app --reload`
- Telegram-бот запускается командой `python -m bot.main`

### Docker Compose

Один запуск `docker compose up --build` поднимает:

- `postgres` - база данных PostgreSQL
- `api` - FastAPI-приложение, которое отдает API, веб-дашборд и админку
- `bot` - Telegram-бот

Особенности Docker-схемы:

- Python-зависимости устанавливаются в образе из `requirements.txt`
- API и бот используют один и тот же код проекта
- каталог `./docs` пробрасывается в контейнеры как `/app/docs`
- автоиндексация включена у `api`
- у `bot` в Compose автоиндексация отключена, чтобы избежать параллельной индексации одних и тех же документов

## 3. Высокоуровневая архитектура

### 3.1. Компоненты системы

Система состоит из следующих логических компонентов:

- `FastAPI backend` в `app/main.py`
- `RAG orchestration` в `app/rag.py`
- `Indexing subsystem` в `app/indexer.py`
- `Search subsystem` в `app/search.py`
- `LLM adapter` в `app/llm.py`
- `Telegram bot` в `bot/main.py`
- `Admin auth` в `app/admin_auth.py`
- `Persistence layer` в `app/db.py` и `app/models.py`
- `Web UI` в `app/static/`

### 3.2. Внешние зависимости

- `PostgreSQL` для хранения документов и чанков
- файловая система для хранения исходных документов
- OpenAI-compatible API для генерации описаний документов и ответов пользователю
- Telegram Bot API для работы Telegram-бота

### 3.3. Взаимодействие компонентов

1. Пользователь взаимодействует с системой через браузер, HTTP API или Telegram.
2. Внешний интерфейс передает вопрос в общий пайплайн `answer_question(...)`.
3. Пайплайн выполняет поиск по БД, собирает контекст и вызывает LLM.
4. Ответ возвращается в исходный канал.
5. Индексация обновляет таблицы `docs` и `chunks`, используя файлы из каталога `docs/`.

Важно: Telegram-бот не вызывает HTTP API. Он использует те же модули приложения напрямую и работает с той же базой данных.

## 4. Модель данных

В проекте используются две основные таблицы.

### Таблица `docs`

Хранит метаданные документа:

- `id`
- `title`
- `description`
- `file_path`
- `file_type`
- `checksum`
- `status`
- `updated_at`

Назначение:

- хранить сведения о документе как о целой сущности
- содержать краткое описание документа, используемое при поиске и генерации ответа

### Таблица `chunks`

Хранит текстовые фрагменты документов:

- `id`
- `doc_id`
- `chunk_index`
- `content`
- `tsv`
- `created_at`

Назначение:

- обеспечивать полнотекстовый поиск
- давать фрагменты контекста для передачи в LLM

### Связь сущностей

- один документ `docs` содержит много чанков `chunks`
- каждый чанк привязан к одному документу
- при переиндексации документа его старые чанки удаляются и создаются заново

## 5. Индексация документов

Индексация запускается несколькими способами:

- при старте API
- при старте Telegram-бота
- через CLI `python -m scripts.index --path ./docs`
- через HTTP `POST /index`
- при загрузке нового документа через админ-панель

### 5.1. Поддерживаемые форматы

- `pdf`
- `docx`
- `dotx`
- `xlsx`
- `xls`
- `csv`

### 5.2. Последовательность индексации

1. Система обходит каталог `DATA_DIR`.
2. Для каждого файла вычисляется `SHA-256 checksum`.
3. Если файл уже есть в БД и checksum не изменился:
   - документ пропускается, если описание уже существует
   - либо обновляется только описание, если оно отсутствовало
4. Для нового или измененного файла извлекается текст:
   - PDF через `pypdf`
   - DOCX и DOTX через `python-docx` или чтение OpenXML
   - XLS/XLSX/CSV через `pandas`
5. Текст нормализуется.
6. Для документа формируется краткое описание:
   - через LLM, если она включена
   - через эвристику, если LLM недоступна
7. Текст режется на чанки функцией `chunk_text(...)`.
8. Чанки сохраняются в таблицу `chunks`.
9. Поле `tsv` строится автоматически на стороне PostgreSQL как `to_tsvector('russian', content)`.

### 5.3. Свойства индексации

- индексация идемпотентна
- исходные файлы не переносятся в БД
- БД хранит только метаданные и текстовые чанки
- повторная обработка неизменившихся файлов не выполняется

## 6. Retrieval и генерация ответа

Общий пайплайн ответа реализован в `app/rag.py`.

### 6.1. Подготовка поисковых запросов

Из вопроса пользователя строится несколько поисковых форм:

- исходный текст вопроса
- очищенный набор токенов без стоп-слов
- дополнительные поисковые варианты для некоторых тем, например для перевода
- отдельные токены длиной от 4 символов

Это повышает шанс найти документ даже при неточной формулировке запроса.

### 6.2. Поиск

Поиск выполняется в `app/search.py` в несколько этапов:

1. `PostgreSQL Full-Text Search` по:
   - `chunks.content`
   - `docs.title`
   - `docs.description`
2. Если FTS не дал результатов, используется fallback-поиск через `LIKE`
3. Если нужно, выполняется дополнительный поиск по отдельным токенам

После этого результаты объединяются, ранжируются и обрезаются до `top_k`.

### 6.3. Сбор контекста

Для LLM формируется:

- список релевантных фрагментов документов
- названия документов
- описания документов
- путь к файлу
- индекс чанка
- краткий обзор библиотеки документов

### 6.4. Генерация ответа

Если LLM включена, системе передаются:

- системный промпт с правилами ответа
- вопрос пользователя
- список извлеченных фрагментов
- краткий обзор каталога документов

Если LLM недоступна:

- возвращается fallback-ответ
- пользователю показываются найденные документы и фрагменты без генеративного синтеза

### 6.5. Выходные данные

Результат работы пайплайна:

- `answer` - итоговый текст ответа
- `sources` - список источников с заголовком, описанием, путем к файлу, индексом чанка и сниппетом

## 7. HTTP API и веб-интерфейс

Основной backend находится в `app/main.py`.

### 7.1. Пользовательские HTTP-эндпоинты

- `GET /health` - проверка доступности сервиса
- `POST /ask` - задать вопрос системе
- `POST /index` - запустить индексацию
- `GET /api/docs` - получить список документов
- `GET /api/docs/{doc_id}/file` - получить или скачать файл документа
- `GET /api/stats` - получить статистику
- `GET /api/history` - получить историю запросов

### 7.2. Веб-страницы

- `/` - пользовательский дашборд
- `/admin` - административная панель

### 7.3. Что делает пользовательский дашборд

- отправляет вопрос к API
- показывает ответ
- отображает найденные источники
- показывает статистику по документам и чанкам
- показывает историю запросов текущего процесса

## 8. Административный контур

Административная логика находится в `app/main.py` и `app/admin_auth.py`.

### 8.1. Что умеет админка

- логин администратора
- просмотр списка документов
- загрузка нового документа
- редактирование заголовка и описания
- удаление документа

### 8.2. Аутентификация

Аутентификация администратора включается, если заданы:

- `ADMIN_USERNAME`
- `ADMIN_PASSWORD`
- `ADMIN_SESSION_SECRET`

Используется подписанная cookie-сессия с TTL 8 часов. JWT или внешняя auth-система не используются.

### 8.3. Что происходит при загрузке документа

1. Загружаемый файл сохраняется в `DATA_DIR`
2. Выполняется `index_file(...)`
3. Документ сразу становится доступным для поиска и ответа

## 9. Telegram-бот

Telegram-бот реализован в `bot/main.py`.

### 9.1. Особенности текущей версии

- бот использует общую бизнес-логику с API
- работает через polling
- может автоматически проверять и индексировать документы на старте
- хранит историю запросов в памяти своего процесса

### 9.2. Структура интерфейса бота

Постоянное меню:

- `Задать вопрос`
- `Документы`
- `Частые темы`
- `Как пользоваться`

Inline-сценарии:

- `Показать источники`
- `В меню`
- `Показать еще` для списка документов
- `Сначала` для возврата к первой странице списка документов

### 9.3. Частые темы

В боте реализованы готовые сценарии по темам:

- перевод
- сессия
- ВКР
- расписание
- физкультура
- устав

По нажатию на тему бот отправляет заранее подготовленный вопрос в общий RAG-пайплайн.

### 9.4. Особенность истории запросов

История запросов хранится в памяти процесса. Это значит:

- API имеет свою историю
- Telegram-бот имеет свою историю
- после перезапуска процесса история очищается

## 10. Текущее эксплуатационное поведение

### 10.1. Типовой запуск через Docker Compose

1. Загружается `.env`
2. Поднимается `postgres`
3. После готовности БД запускается `api`
4. API проверяет схему БД и выполняет автоиндексацию
5. После готовности API запускается `bot`
6. Бот начинает polling Telegram Bot API

### 10.2. Что реально считается фронтендом

Отдельного frontend-сервиса в проекте нет. Статические файлы интерфейса расположены в `app/static/` и обслуживаются тем же контейнером `api`, который отдает FastAPI-приложение.

## 11. Основные преимущества выбранной архитектуры

- единая кодовая база для API, веб-интерфейса и Telegram
- отсутствие дублирования RAG-логики
- простая схема хранения `docs + chunks`
- идемпотентная индексация
- минимальная инфраструктурная сложность
- возможность запуска одной командой через Docker Compose
- понятный путь дальнейшего расширения

## 12. Ограничения текущей версии

- история запросов хранится только в памяти процесса
- поиск основан на FTS и не использует embeddings
- качество ответа зависит от качества извлечения текста из PDF и таблиц
- админская аутентификация реализована через подписанную cookie, без полноценной IAM-системы
- Telegram-бот и API разделяют БД, но не разделяют in-memory историю
- LLM является внешней сетевой зависимостью

## 13. Направления развития

- добавить векторный поиск и гибридный retrieval
- перенести историю запросов в БД
- добавить отдельную пользовательскую память по сессиям
- улучшить парсинг табличных и PDF-документов
- расширить административную панель
- добавить более точные health-check и диагностику LLM
- расширить сценарии Telegram-бота

## 14. Краткий вывод

В текущем состоянии проект представляет собой централизованную RAG-систему с единым прикладным контуром и несколькими интерфейсами доступа. Архитектура уже покрывает полный цикл работы с локальной базой документов: загрузку, индексацию, поиск, генерацию ответа, веб-доступ, администрирование и Telegram-интерфейс. При этом система остается достаточно компактной, чтобы ее можно было сопровождать и дорабатывать без перехода к сложной микросервисной схеме.
